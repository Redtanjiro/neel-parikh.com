**Accessibility:** each card is a single `<a class="work-card">` wrapping the image, index, title, description, and the visible "View case study →" affordance (a plain `<span>`, not a nested link) — one tab stop per card, confirmed by construction (no second focusable element inside). The CTA's hover-draw underline triggers off `.work-card:hover`/`:focus-visible` (parent-scoped), not a self-scoped `.link-draw`, specifically so hovering anywhere on the card (image, title, description) shows the affordance — not just hovering the small CTA text itself.

**Filter-chip findings (per the spec's explicit ask, reported not silently changed):**
- **No filter is active by default on load.** `currentFilter` initializes to `null`, `.filter-chip` has no `.is-visible` class until a discipline button on the hero is clicked. Confirmed by reading the initialization code, not a runtime check.
- **No card markup exists pre-JS.** `#work-cards` is an empty `<div>` in the raw HTML; all four cards are built by `featured.forEach(...)` at script-run time, reading from `window.PROJECTS`. A crawler that doesn't execute JavaScript would see zero project cards — this is a pre-existing characteristic of the site's single-source-of-truth data architecture (unchanged from the old accordion, not introduced by this build), not something I changed. Flagging per the spec's ask, not fixing — a real fix would mean static-rendering the cards server-side or at build time, which conflicts with this project's explicit "no build step" constraint (`CLAUDE.md` non-negotiable #2) and is a bigger architectural call than this spec asked for.
- **No GSAP Flip-plugin reflow exists** — never built (confirmed via `grep -n "Flip"`, zero hits). The filter simply toggles `.is-hidden` (`display:none`) on non-matching cards, same as the old accordion did. Flip wasn't part of the shipped site before this spec either, so nothing was removed — just noting it's absent, per the spec's ask to check.

**Motion:** used the same restrained fade+rise-once reveal already shipped elsewhere on this site (`opacity`/`y` from a `ScrollTrigger` with `once: true`, `start: 'top 80%'`, slight per-card stagger) rather than inventing "V2 ink-reveal," since that term has no definition available anywhere in the repo. Hover: `.work-card-media img` scales to `1.03` under `@media (hover: hover)`, transform-only, 0.2s. Reduced-motion branch: the scroll-reveal script block is skipped entirely under `prefers-reduced-motion: reduce` (cards render at their default, fully-visible state, no animation), and the hover-scale transform is explicitly neutralized in the same media query.
