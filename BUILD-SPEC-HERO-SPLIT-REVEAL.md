## Resolved before building

**Biggest open question in the spec — asked Neel directly:** adopt the reference's fixed left-sidebar two-column layout, or keep the current top-nav-pill single-column layout? **Answer: keep the current layout.** No sidebar rebuild — this spec is scoped to the hero→About→Work scroll sequence only, layered onto the existing page skeleton.

**What "splits" — resolved during build, not literally everything in `#hero`.** The spec's reference is a single illustration splitting into two curtain-halves. This site's hero also contains interactive UI on top of the illustration (discipline buttons with hover rings, the clock, cursor-driven pose tracking) — duplicating those into two halves would either break `id`-based JS (duplicate `id`s silently break `getElementById`, per `CLAUDE.md` trap #4) or require rewriting working interactive systems for no visual benefit. So: **only the illustration layer splits** (`.hero-scenes` + `.hero-poses` + `.hero-dim`, duplicated into `.hero-half--left`/`.hero-half--right`, each clip-pathed to one half with a small overlap at the seam per the spec's own flicker warning). `.hero-copy`, `.hero-disciplines`, the clock, and the scroll-cue stay as single, un-duplicated elements that fade out over the same scroll range — not physically split.

**Reveal mechanism — `pin: true, pinSpacing: false` on `#hero`**, exactly as the spec's own notes call out ("`pinSpacing: false` is what lets About slide up underneath... the key property"). `#hero` gets a `z-index` above `#about` so it visually covers About until the halves clear. **This alone was not sufficient — see "Build notes: the black-gap bug" below for what `pinSpacing:false` actually does and doesn't fix, and the real cause/fix.**

**About redesign scope.** New display-type lead-in line (composed from Neel's identity, mixed serif-italic/heavy-sans emphasis per the reference), each line staggers in, a hand-drawn squiggle underline (site's existing `stroke-dashoffset` technique, not a new plain underline) draws in under "illustrator." The previously-approved paragraph (unchanged, see below) sits beneath as calmer supporting copy — a simple fade/rise, not per-line staggered, since it's supporting text, not the display statement.

**Work section — scope-limited to the safe half of the ask.** The spec describes the reference's own card format ("big serif+italic headline + an outcome/stat line") alongside the actual action item ("standard scroll-reveal as each card enters viewport"). Implementing a full redesign into a stat-line card format is a bigger, separate content decision — not all case studies have a real, honest metric (`CLAUDE.md`'s case-study rule #4: never fabricate one; Futee is a Figma prototype with no conversion rate). **Built: scroll-reveal (fade + y-offset) added to the existing accordion rows, format unchanged.** Flagged to Neel as a separate open question, not assumed.

**Fraunces re-added to the Google Fonts import.** Dropped when the frosted-glass intro was removed (its only consumer at the time); About's new mixed-emphasis display line is a legitimate About-page use per `CLAUDE.md`'s existing carve-out ("Fraunces and Caveat exist but are About-page only").

## Build notes: the black-gap bug (reported after first build, fixed same session)

Neel scrolled the first build in his own browser and found a full-viewport black band between the nav pill and About/Chosen Work — confirmed it scrolled *with* the page, not a stuck pin. Two separate real bugs, both fixed:

**1. `pinSpacing:false` does not do what the name suggests.** Confirmed via GSAP's own docs: it only skips reserving *extra* space *during* the pin. It does **not** remove the pinned element's own natural height from the flow *after* it unpins — `#hero`'s own `min-height:100vh` box still occupied a full viewport of document flow once the pin released, before `#about` (a normal sibling below it) could appear. That resting-height gap was the actual black band.

Fix: **`#hero` and `#about` now share one `.hero-stage` container** (`position:relative`). `#hero` is `position:absolute; top:0; left:0; width:100%; height:100vh` inside it — contributing **zero** flow height, ever, regardless of pin state. `#about` (normal flow, tall — paragraph, credential, tools) is what actually determines `.hero-stage`'s height. This is the pattern GSAP's own docs recommend for pinSpacing:false ("only when the pinned element is position:absolute or otherwise removed from flow") — confirmed by an explicit search of GSAP's documentation before committing to it.

Making `#hero` absolute exposed a second GSAP quirk: with only `left/right` (no explicit `width`), GSAP's pin-spacer measured it as **zero width** at pin setup. Fixed with an explicit `width: 100%` instead of `left:0;right:0`.

**2. `#hero`'s own background never faded.** Only the split halves' *content* (the illustration) animated — `.hero`'s own `background: var(--ink)` fill covers its whole box regardless, and nothing was fading that box itself. Once unpinned, that still-opaque, still-z-index:10 box kept covering whatever was genuinely underneath for the ~900px it took to scroll past — visually indistinguishable from a black gap, even after fix #1. **Fixed by fading `#hero` itself (`autoAlpha`, the whole section, not just its children) to invisible by the end of the split timeline.**

**3. Found during verification, not reported by Neel: the About statement was revealing itself off-screen.** Since `#about` sits in normal flow behind the pinned hero, it scrolls at the ordinary 1:1 rate throughout the pin. Measured `#about-statement`'s own document position (`top: 127.5px` within `#about`) against when the line-reveal starts (scroll ≈270px, per the timeline's `0.3` position parameter) — the statement had already scrolled ~140px above the viewport *before the reveal even began*. The opacity math was correct (confirmed numerically) but invisible, because the element wasn't on screen to see it happen. **Fixed with `position: sticky; top: 110px` on `#about-statement`** (mirrors the site's existing `scroll-padding-top: 100px` nav-offset convention), so it holds its position long enough to actually play its own reveal, then releases into normal flow as `#about`'s remaining height (paragraph, credential, tools) runs out. Disabled on mobile (`position:static`) alongside the rest of the split effect.

**Verification caveat:** `position:sticky` is confirmed via `getComputedStyle` (correct `position: sticky; top: 110px`) but **cannot be visually verified in the Claude Code Browser pane** — `CLAUDE.md` trap #13 already documents that this tool's compositor never visually clamps sticky elements, regardless of trigger, even when the CSS is correct. This one needs confirming in Neel's own browser.

## Build notes: the black-panel bug (reported with real screenshots, fixed same session)

Neel supplied four real-browser screenshots showing the split working but revealing an opaque **black** panel instead of About, and About rendering dark-on-dark once revealed. A computed-style check on `#about` alone (paper bg, ink text — both correct) couldn't explain it, because it never asks what's *painted on top of* `#about`. Confirmed the actual cause two ways: static inspection found `.hero` still carried `background: var(--ink)` at `z-index:10` — its own box doesn't move when its children (the halves) translate apart, so for most of the split it kept covering `#about` regardless. `document.elementsFromPoint(x, y)` at mid-split confirmed it directly: `SECTION#hero` was the first opaque layer in the paint stack, sitting right above `#about`. **Fix:** moved the dark fallback background off `.hero` entirely, onto `.hero-half` (both halves) instead — each half now carries its own solidity as it translates, so the vacated centre strip is covered by nothing once both have moved. Added CLAUDE.md trap #14 for this: a passing computed-style check on the target element proves nothing about what's painted on top of it; visual bugs from a real browser need a paint-stack check (`elementsFromPoint`), not just re-confirming the target's own styles.

## Build notes: the recompose pass (2026-07-23, same day)

Split and reveal both worked, but the *composition* didn't read as two clean "screens" — hero text lingered, About's line-reveal and the supporting paragraph weren't sequenced against each other, and the statement wasn't positioned deliberately. Neel asked for explicit checkpoints (25/50/75/100% of the pin) that each had to look like one complete, non-overlapping state, plus a real layout fix: the statement vertically centred in the pinned viewport, and the supporting paragraph/tools group placed and revealed strictly after the statement, never before or above it.

**Markup restructured into two groups inside `#about-content`:**
* `#about-reveal` — eyebrow + `#about-statement`. `position: sticky; top: 0; height: 100vh; display:flex; flex-direction:column; justify-content:center; padding-top: 110px;` — centres the statement in the pinned viewport at every progress point, and the `padding-top` guarantees it clears the nav pill regardless of `#about`'s total height. Held at `opacity:0; visibility:hidden` by default; the split timeline flips it visible at progress 0.45. Carries its own opaque `background: var(--paper)` for the same reason `.hero-half` needed one (trap #14) — a sticky element with no background of its own lets whatever scrolls underneath bleed through it.
* `#about-support` — paragraph, credential, résumé link, tools row, grouped as one unit. Held at `opacity:0` and revealed only by its own trigger *after* the pin ends (see below) — never during it.

**Real bug: the paragraph could appear before the statement had revealed.** Previously `.about-copy` had its *own*, independent `ScrollTrigger` keyed to its own scroll-into-view position, entirely decoupled from the split timeline. Nothing tied the two together, so the paragraph's trigger could fire while the statement's lines were still at opacity 0. Fixed by making `#about-support` a single grouped reveal keyed to `trigger: heroEl, start: 'bottom top'` — this fires at exactly the scroll position where the pin's own range ends (computed correctly by ScrollTrigger even while hero is pinned/fixed), so it can never begin before the split's 100% checkpoint.

**Real bug, more fundamental: GSAP timeline position labels are TIME, and scrub/`.progress()` map scroll-fraction onto a fraction of the timeline's TOTAL duration — not directly onto the authored position numbers.** An interim version's tweens summed to a total duration of 0.9 (not 1.0), so authored positions silently drifted — "0.45" actually fired at real scroll-progress 0.5, "0.7" at ~0.78, etc. Invisible by eyeballing scrub'd scroll behaviour; only surfaced by driving `tl.progress(p)` directly (bypassing scrub smoothing) and comparing the result against the authored labels. **Fix: chose every tween's start/duration so the *latest* one ends at exactly 1.0** (confirmed via `tl.totalDuration() === 1`), then re-verified every checkpoint both via `tl.progress()` and via real `scrollTo()` + `ScrollTrigger.update()` — identical results at 0.25/0.5/0.75/1.0. Also dropped the `stagger` shorthand for the three `.about-line` tweens in favour of three explicit `.to()` calls with named positions — combined with explicit position parameters, `stagger`'s actual start/duration didn't land where the numbers implied, and named positions removed the ambiguity entirely rather than reverse-engineering GSAP's interaction between the two.

**Final checkpoint timeline** (total duration exactly 1.0):
* `[0, 0.15]` — hero copy/chips/clock/scroll-cue fade to 0
* `[0.1, 0.7]` — halves part (`xPercent: ±65`)
* `0.45` — `#about-reveal` flips visible (instant `.set()`, nothing to see yet — the lines are still their own opacity:0)
* `[0.45, 0.6]`, `[0.55, 0.7]`, `[0.6, 0.75]` — the three `.about-line`s, explicit per-line tweens
* `[0.75, 0.9]` — squiggle draws under "illustrator" (line 2, already done revealing by 0.75)
* `[0.9, 1.0]` — `#hero` itself fades to `autoAlpha:0` (cleanup — halves and hero-copy are long gone by this point, just avoids a lingering invisible-but-interactive element at z-index:10)
* `#about-support` reveals only once scrolled *past* 1.0 (`start:'bottom top'`), never during

**Real bug found in verification, not reported:** `#about-support`'s reveal `ScrollTrigger` wasn't gated to desktop — GSAP applies an inline `opacity:0` the instant a scrub `ScrollTrigger` is created, on every viewport width, which silently overrode the mobile CSS fallback (inline styles beat stylesheet rules regardless of media query). Fixed by adding the same `min-width:900px` gate the rest of the split effect uses.

**Second bug found in the same mobile check, unrelated to the first:** even after removing the inline-style override, `#about-support` still computed `opacity:0` on mobile. Cause: the mobile-fallback rule (`.about-support{opacity:1}`, inside the `@media (max-width:899px)` block) was declared *before* the base rule (`.about-support{opacity:0;...}`) in the stylesheet — with equal specificity, source order decides, so the later base rule silently won regardless of the media query matching. Fixed by moving the base rule above the media query block.

**Verification:** all four checkpoints (25/50/75/100%) checked via `tl.progress()` directly and cross-confirmed via real `scrollTo()` + `ScrollTrigger.update()` — identical results both ways. Reverse (back to 0) restores full rest state cleanly. Zero console messages throughout, on both desktop and mobile widths. Screenshots at 50%/75%/100% hit the pre-existing, already-documented Browser-pane blank-capture quirk (trap #11) — cross-checked with `elementsFromPoint` paint-stacks and `get_page_text` at each of those points, both confirming the content is correctly present, ordered, and unobscured; the 25% checkpoint's screenshot rendered cleanly and shows the intended composition (parted halves, clean paper gap, zero hero-text ghosting).
